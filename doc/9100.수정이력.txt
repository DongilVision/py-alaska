■ InvokeDispenser CPU 절감 최적화 (4대 최적화)

1. RMI 응답 대기 — Spin-wait → Kernel Event (task_manager.py)
   - 변경 전: RmiClient._call()이 response_q.read()를 폴링 (CPU 1코어 점유)
   - 변경 후: response_event.wait(timeout)로 커널 대기 (CPU 0%)
   - 추가 자원: 태스크당 response_event (SmKernelEvent)
   - InvokeDispenser: 응답 기록 후 caller_ti.response_event.set()
   - 수정 파일: task_manager.py (TaskInfo, RmiClient, InvokeDispenser, TaskRuntime)

2. Signal Direct Emit — SEND 위임 제거 (task_signal.py)
   - 변경 전: emit() → send_queue → InvokeDispenser ① SEND → broker.emit()
   - 변경 후: emit() → broker.emit() (caller 스레드에서 직접)
   - SEND 단계 삭제. send_queue 제거. 디스패처 루프 4단계 → 3단계
   - 수정 파일: task_signal.py (SignalClient), task_manager.py (InvokeDispenser)

3. 조건부 notify — is_sleeping 플래그 (task_manager.py, task_signal.py)
   - 변경 전: 모든 시그널/RMI 전송마다 notify_event.set() (커널 syscall)
   - 변경 후: is_sleeping == 1일 때만 set() 호출 (메모리 읽기 ~50ns)
   - 추가 자원: 태스크당 is_sleeping (SmValue 'B', 1바이트)
   - 수정 파일: task_manager.py, task_signal.py (SignalBroker._notify_task)

4. 시간 호출 간소화 (task_manager.py)
   - 변경 전: RECV burst guard에 perf_counter() 호출 (최대 38회/cycle)
   - 변경 후: 카운트 기반 burst guard (50건/cycle, 시간 호출 0회)

■ 성능 검증 중 발견·수정 사항

   (a) get_client() — notify_event 미전달 → 외부 RmiClient에 notify_event 전달 추가

   (c) 자식 프로세스 — 타 태스크 response_event 미재연결 → 모든 태스크 재연결
   (d) _sm_registry_name None 방어 → (... or '') 방어 코드 추가

■ InvokeDispenser — IPC/Signal 통합 단일 스레드 디스패처

1. InvokeDispenser 도입 (task_manager.py)
   - 기존 _rmi_handler(RMI 전용 스레드) + SignalClient._recv_loop(시그널 수신 스레드)
     → InvokeDispenser 단일 스레드로 통합
   - Signal 우선 → IPC 후순 → Kernel Wait 3단계 루프
   - 태스크당 스레드 2개 (Dispenser + Main) — 기존 3개에서 절감
   - Deadlock-Free: chain 기반 순환 호출 감지, Self-call 감지

2. SmKernelEvent 기반 커널 대기 (sm_kernel_event.py)
   - Win32 CreateEventW/WaitForSingleObject 래퍼
   - CPU 0% 대기, 20~50μs 반응 속도
   - notify_event: 태스크당 1개, 모든 Signal/RMI 수신 통지 공유

3. SignalClient headless 모드 (task_signal.py)
   - InvokeDispenser가 시그널 수신을 담당하므로 _recv_loop 스레드 미시작
   - emit() → send_queue 위임 (InvokeDispenser ① SEND 단계에서 처리)

4. PriorityScheduler — QoS 기반 스케줄링 (task_manager.py)
   - Weighted Fair Queuing + Starvation Guard
   - CYCLE_QUOTAS = [20, 10, 5, 2, 1] (P1~P5)

■ 설계서: doc/2016.InvokeDispenser설계.txt

■ Signal QoS 및 SmInfra 통합

1. QoS 5단계 우선순위 (task_signal.py)
   - P1(CRITICAL), P2(REALTIME), P3(NORMAL), P4(LOW), P5(BACKGROUND)
   - Deadline 위반 감지 및 자동 폐기
   - QoS 프로필: 딕셔너리 또는 문자열로 지정

2. SmSignalRegistry — 공유 메모리 구독 레지스트리 (sm_signal_registry.py)
   - Manager.dict IPC 프록시 제거 → SharedMemory 비트맵 직접 접근
   - O(1) 구독자 조회 (sub_mask 비트 연산)
   - emit() hot-path에서 Manager IPC 호출 0회

3. SmSignalQueue — MPSC 시그널 큐 (sm_infra/sm_queue.py)
   - SmRingBuffer + Named Mutex (WinMutex)
   - 다중 Producer 안전, pickle 기반 메시지 전달

4. SmSignalStats — 실시간 통계소 (sm_signal_stats.py)
   - 시그널별 Transit/Proc Latency EMA 집계
   - 웹 디버거 실시간 차트 연동

5. SmBlackBox — 시그널 체인 세션 감시 (sm_blackbox.py)
   - 체인(A→B→C) 완료 보장, 타임아웃 시 경보 발생

6. Watchdog 자가 치유 (task_manager.py)
   - 하트비트 기반 태스크 정지/좀비 감지
   - 자동 재시작 (restart=True 설정 시)

■ 설계서: doc/2013.시그널QOS기능V2.txt, doc/2014.SmInfra설계.txt

■ Signal 시스템 안정화 및 _seq 트래킹 추가

1. Signal 객체 재사용 버그 수정 (task_signal.py)
   - 증상 : "Block X is already free" 오류, SmBlock 이중 해제
   - 원인 : _recv_loop에서 Signal 객체 1개를 재사용하는 구조에서,
            @ui_thread 데코레이터가 클로저를 Qt 이벤트 큐에 지연 등록하는 동안
            Signal 객체의 데이터가 다음 시그널로 덮어씌워짐
   - 수정 : _recv_loop에서 수신 메시지마다 Signal 객체를 새로 생성하도록 변경
   - 파일 : src/py_alaska/task_signal.py (SignalClient._recv_loop)

2. 플랫폼 레벨 _seq 시퀀스 트래킹 추가 (task_signal.py)
   - 목적 : 시그널 누락(drop) 여부를 수신측에서 자동 감지
   - 동작 :
     · 송신측(SignalClient.emit)에서 signal_name별 seq 번호 자동 부여 (1부터 증가)
     · seq 번호는 전달 경로 전체를 통해 보존됨
       (emit_fast → relay_queue/shm_buffer → _distribute_signal → subscriber_queue)
     · 수신측(_recv_loop)에서 (source, signal_name) 키로 마지막 seq를 추적하여
       gap 발생 시 표준 출력에 경고 메시지 출력
       예: [Signal:viewer] 2 missed: 'camera.received' from 'camera1' (seq 3~4)
   - 하위호환 : _seq=0(기본값)인 경우 트래킹 비활성화, 기존 코드 영향 없음
   - 수정 파일 :
     · src/py_alaska/task_signal.py
       - SignalClient.__slots__      : _emit_seqs, _recv_seqs 추가
       - SignalClient.__init__       : seq dict 초기화
       - SignalClient.emit()         : signal_name별 seq 증가, emit_fast에 전달
       - SignalClient._recv_loop()   : gap 감지 로직 추가
       - SignalBroker.emit()         : _seq 파라미터 전파
       - SignalBroker.emit_fast()    : _seq 파라미터 전파
       - SignalBroker._distribute_signal() : _seq 파라미터 전파
       - SignalBroker._relay_loop()  : data.get('_seq', 0) 전달
       - SignalBroker._relay_loop_shm() : data.get('_seq', 0) 전달
       - SignalSharedBuffer.put()    : _seq 파라미터, pickle payload에 포함

3. 카메라 프레임 전달 방식 변경 (ex130_camera)
   - 변경 전 : nextTask.on_input.nowait(frame_info) — RMI 1:1 전달
   - 변경 후 : signal.camera.received.emit({...})  — Signal 브로드캐스트
   - 효과 : 여러 구독자가 동시에 카메라 프레임을 수신 가능
   - 수정 파일 :
     · src/py_alaska/drives/imi/imi_camera.py   : emit 방식으로 변경
     · example/ex130_camera/task_viewer.py      : on_input → on_camera_received
     · example/ex130_camera/config.json         : nextTask 제거, 클래스명 대소문자 수정

4. 성능 테스트 기준값 조정 (ex140_performance)
   - 변경 전 : SIGNAL_TARGET_MS=0.1, IPC_TARGET_MS=0.1, TARGET_TPS=1000 (비현실적)
   - 변경 후 : SIGNAL_TARGET_MS=0.5, IPC_TARGET_MS=3.0, TARGET_TPS=300 (실측 기반)

■ 최초 릴리즈

   - TaskManager : thread/process 기반 태스크 생성·관리
   - RMI         : Queue 기반 원격 메서드 호출 (동기/비동기)
   - Signal      : Pub/Sub 브로드캐스트 이벤트 시스템
   - SmBlock     : 공유메모리 블록 풀 (zero-copy 프레임 공유)
   - GConfig     : JSON 기반 계층형 설정 관리
   - @task       : on_* 메서드 자동 시그널 구독 데코레이터
   - @ui_thread  : Qt 메인 스레드 전환 데코레이터
